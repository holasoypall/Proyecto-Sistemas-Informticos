The journey: From Scrath to the Operating System

  1. Phase 1: Ignition Self-Diagnosis (POST)

    As soon as power is supplied, the CPU initializes itself and its Program counter(PC) is set to a previously established memory address. This addres points to the firmware ROM (BIOS or UEFI). The first instruction executed is the POST (Power-On Self-Test), which is an in-place diagnostic that checks if the important hardware is correct:

    • Testing the CPU (records, flags).
    • Testing the RAM (detection and basic integrity test).
    • Display output.
    • Testing main buses (PCI, PCIe).

    An error in this sequence will prevent the computer from booting, it will be communicated with beeps (POST codes).

  2. Phase 2: Firmware Initialization and Device Selection

    Once the POST has been completed, the firmware initializes low-level drivers and displays the devices. Then, it checks the configuration in non-volatile memories (NVRAM, formerly CMOS) to determine the boot sequence (Boot Priority). The ordered sequence (e.g., 1. NVMe SSD, 2. USB, 3. PXE Network) tells the firmware where to look for the boot code.

  3. Phase 3: Carga del Código de Arranque (Boot Code Loading (MBR vs. UEFI Fork))

    This is the point where the traditional and modern starting methods differ:

    • Legacy Mode (BIOS/MBR): The BIOS reads the first logical sector (Sector 0, 512 bytes) of the Master Boot Record (MBR). The BIOS loads these 512 bytes into a physical memory address and transfers control to the CPU. This action loads the next stage of the bootloader.
    • Current Mode (UEFI/GPT) The UEFI, acting as a mini operating system, has the ability to work with files (e.g., FAT32). It does not read a fixed sector but looks for a specific partition called GUID: the EFI System Partition (ESP). Within the ESP, it navigates through a directory and executes an EFI file.

  4. Phase 4: Boot Manager Execution (Stage 2)

    Now we are at the stage of the main boot manager such as GRUB2, or Windows Boot Manager, which is already loaded into RAM. This software is a bit more complex:

    1. Interprets its own configuration file.
    2. Reads multiple file systems such as ext4 or NTFS.
    3. It has an interactive menu so the user can differentiate between different operating systems or kernels.
    4. Locate the kernel image and the initial RAM disk in the file system.

  5. Phase 5: Loading the Kernel and Initial RAM Disk (initrd)

    When the user selects an entry (or by default), the boot manager loads two items into RAM:

    • The Kernel: The operating system kernel image. It is responsible for managing memory, processes, and hardware.
    • The `initrd` (initial ramdisk): It is a file that contains a temporary root filesystem. Its purpose is to manage the modules and drivers that the kernel needs to mount the actual root filesystem (different from the one that has already been created).

  6. Phase 6: Kernel Initialization and Real Root Assembly

    Control of the computer passes to the kernel. It decompresses itself, initializes (configuring memory management, interrupts, etc.), and mounts the "initrd" as its temporary root system ("/") so that it can mount the real root later.

    From this temporary root, the kernel loads the "initrd" modules, detects the actual hard drive, and mounts the permanent root filesystem. Finally, it uses the "pivot_root" (or "switch_root") system call to discard the initrd and adopt the permanent filesystem as its new root directory.

  7. Phase 7: Execution of the "init" Process (User Space)

    With the actual file system created, the kernel has completed its boot process. Its final task is to execute the first user-space process.

    In modern Linux systems, this is `systemd`. This process (which is no longer the kernel) becomes the parent of all system processes. It reads its own configuration and simultaneously starts the rest of the system: network files, mounting secondary file systems, and finally, the graphical environment and login system. In this way, it presents the welcome screen to the user.
